# Othello Application Architecture & Logic Reference

このドキュメントは、Othelloアプリケーションのデータベーススキーマ、エンジン、サービス層、API層、およびAI/DQNロジックの依存関係と全体像をまとめたものです。
リファクタリング、デバッグ、アップデート、およびAIによるコード生成のコンテキストとして活用してください。

---

## 1. コアエンジン層 (`othello_env.py`)

オセロの完全なルールとドメイン知識をカプセル化した純粋なPythonクラス（副作用なし）です。
盤面は8x8の2次元配列 (`List[List[int]]`) として表現されます。

* **石を裏返す判定 (`_get_flippable_discs`)**: 8方向探索を用い、「隣が相手の石であり、その先に自分の石がある」というオセロ特有のサンドイッチ条件を厳密に判定します。
* **状態遷移 (`apply_move`)**: 副作用を防ぐため、常に現在の盤面のディープコピーを作成してから石を置き、裏返して新しい盤面を返します。
* **パス判定 (`has_valid_moves`)**: プレイヤーに合法手が残っているかを計算し、連続パスやゲーム終了条件の検知に利用されます。

## 2. データベース・モデル層 (`models.py`)

Django ORMを使用して定義され、データの永続化を担います。静的型付けツール(mypy)に耐えうるよう、各メソッドやバリデータには厳格な型ヒントが付与されています。

### 2.1 `GameSession` モデル
進行中・完了済みのオセロのゲーム状態を管理します。

* **`id`**: UUID (Primary Key)。推測不可能なURL/APIキーとして機能します。
* **`user`**: `User` モデルへの外部キー。
* **`opponent_type`**: 対戦相手の種類 (`"ai"` または `"random"`)。
* **`user_color`**: ユーザーの石の色 (`1`: Black/先手, `-1`: White/後手)。
* **`current_board`**: 8x8の2次元配列を格納する `JSONField`。カスタムバリデータ (`validate_board_format`) で厳密に型と構造が検証されます。
* **`current_turn`**: 現在の手番 (`1` または `-1`)。
* **`status`**: ゲームの進行状況 (`"playing"`, `"finished"`, `"abandoned"`)。
* **タイムスタンプ**: `created_at`, `updated_at` (自動更新)。

### 2.2 `MatchHistory` モデル
対戦結果の履歴を管理します。ユーザーごとに最大10件（`MAX_MATCH_HISTORY`）のみ保持されます。

* **`user`**: `User` モデルへの外部キー。
* **`opponent_type`**: `GameSession` と同様。
* **`result`**: 勝敗 (`"win"`, `"loss"`, `"draw"`)。
* **タイムスタンプ**: `played_at`

## 3. サービス層 (`services.py`)

Webリクエスト（View層）とデータ・環境操作（Model/Env層）を繋ぐビジネスロジックの中核です。全関数に `@transaction.atomic` が付与され、データベースの整合性を担保しています。

### 3.1 排他制御とバリデーション (Optimistic Locking)
* **`process_move`**: 石を置く際の最重要処理。
  1. **状態確認**: セッションが `PLAYING` であるかをチェック。
  2. **排他制御 (TurnConflictError)**: リクエストの `expected_turn` とDBの `current_turn` が一致するか確認。連打や複数ウィンドウによる状態不整合を防ぎます。
  3. **合法手チェック (InvalidMoveError)**: `OthelloEnv.is_valid_move` でルール違反がないか確認。
  4. **盤面更新**: `OthelloEnv.apply_move` と `change_turn` を呼び出し、DBへ保存。

### 3.2 セッションと履歴のライフサイクル管理
* **`start_game`**: 進行中のゲーム（`PLAYING`）を検知した場合、自動的に `ABANDONED`（放棄・敗北扱い）にしてから新ゲームを作成します。ユーザーの色はコイントス（ランダム）で決定されます。
* **`surrender_game` / `process_timeout_abandoned_games`**: ユーザーの明示的な降伏、または1時間以上の放置（タイムアウト）時にセッションを終了し、敗北履歴を記録します。
* **`save_match_history`**: 履歴追加時に、最新10件をクエリで特定し、それより古いレコードを一括削除（クリーンアップ）します。

## 4. API / コントローラー層 (`views.py`)

フロントエンドからのRESTfulなリクエストを受け付け、サービス層へ処理を委譲するオーケストレーターです。`LoginRequiredMixin` により全エンドポイントが保護されています。

### 4.1 エンドポイント
* **`POST /api/game/start/`**: ゲームを開始します。AI先手の場合は、レスポンスを返す前にAIの手番を処理します。
* **`POST /api/game/move/`**: ユーザーの手を処理します。ユーザーの手の直後に、必要に応じてAIの手番を連続して処理します。
* **`GET /api/user/history/`**: ユーザーの過去の対戦履歴（最大10件）を取得します。

### 4.2 特徴的なオーケストレーションロジック
* **非同期不要の高速AIターン (`_check_and_run_ai_turns`)**: 
  オセロにおける複雑な「連続パス」のエッジケースを `while` ループを用いてバックエンド内で解決します。
  * **ユーザーパス**: ユーザーに合法手がない場合、APIはエラーを返さず自動でAIに手番を渡し、AIの手を実行した後の盤面をフロントエンドに返却します。
  * **AIパス**: AIに合法手がない場合、AIの手番をスキップし、ただちにユーザーに手番を戻します。
  * これにより、フロントエンドは「複雑なパス管理」を意識することなく、常に「自分が操作可能な状態」あるいは「ゲーム終了状態」の盤面を受け取ることができます。

### 4.3 堅牢なエラーハンドリング
例外の型に応じて適切なHTTPステータスコードを返し、クライアント（フロントエンド）に正確な状況を伝達します。
* **400 Bad Request**: ペイロード不足、無効なUUID形式、非合法手 (`InvalidMoveError`)、終了済みゲームへの操作、JSONパースエラー。
* **404 Not Found**: 存在しないセッションID。
* **409 Conflict**: ターン不一致 (`TurnConflictError`)。フロントエンド側の状態更新漏れやダブルクリック検知。
* **500 Internal Server Error**: その他の予期せぬ例外。

## 5. AI / DQNロジック (`Agent.py`, `dqn_model.py`)

PyTorchで実装された深層強化学習（Distributional DQN）ベースのAIモデルです。

### 5.1 ネットワークアーキテクチャ (`dqn_model.py`)
* **SEResNetBlock (Squeeze-and-Excitation)**: 特徴量のチャネル間の依存関係を学習できるSEアテンション機構を取り入れたResNetブロックを採用しています。
* **Dueling Network + C51**: 
  * 状態価値（Value）と行動利得（Advantage）を別々に計算して合成します。
  * 出力は単一のQ値ではなく、**C51（カテゴリカルDQN）**に基づく51個のAtom（確率分布）です。出力Shapeは `(Batch, 64, 51)` となります。

### 5.2 AIエージェントロジック (`Agent.py`)
AIの推論を安全かつルールに沿って実行するためのラッパーです。

* **Observation生成 (`_get_observation`)**:
  8x8の盤面を、DQNが理解しやすい `(3, 8, 8)` のテンソル（特徴量マップ）に変換します。
  * **チャンネル0**: 自分の石 (1.0 or 0.0)
  * **チャンネル1**: 相手の石 (1.0 or 0.0)
  * **チャンネル2**: 合法手の位置フラグ (1.0 or 0.0)
* **推論とAction Masking (`get_move`)**:
  1. Observationをネットワークに入力し、確率分布（`q_log_probs`）を取得。
  2. `V_MIN` から `V_MAX` の範囲（51等分）のサポート値と確率を内積し、各マスの期待値（Q値）を算出。
  3. **合法手マスキング**: ルール違反を防ぐため、**合法手でないマスのQ値を強制的に `-1e10` に設定**します。ネットワークが未熟でも非合法手を選ぶことはありません。
  4. 最大Q値のインデックスから、行（Row）と列（Col）を算出。
* **耐障害性**: `__init__` での重みロード時に例外をキャッチする構造を持ち、ファイル破損等でロードに失敗してもクラッシュせず、ランダム初期化された重みで安全に起動を継続します。（セキュリティ強化のため `weights_only=True` オプションを適用済み）。

## 6. アーキテクチャの品質保証 (Quality Assurance)

* **静的型解析 (mypy)**:
  `--strict` モード相当の強力な型チェックに耐えうる設計です。
  * Django ORMに由来する動的型付け（例: `request.user` の `User | AnonymousUser` 問題）に対しては、`typing.cast` を用いて厳格に型を解決しています。
  * ドメインモデルやサービス層の全関数の引数・戻り値には、`List`, `Dict`, `Tuple` 等を用いた明確な型シグネチャが定義されています。
  * `Any` や `# type: ignore` を極力排除し、実行時の型エラーをコンパイルレベルで防御します。

* **テスト駆動開発 (TDD)**:
  Pytestによる網羅的なテストスイート（`tests/`）が構築されています。
  * **ドメイン層**: `OthelloEnv` の複雑な盤面バリデーション、裏返し判定を境界値含めてテスト。
  * **サービス層**: DBのトランザクション、楽観的ロック（ターンコンフリクト）の例外発生を検証。
  * **ビュー/API層**: JSONフォーマット、異常系HTTPステータス (400, 404, 409)、連続パスターン消費のエッジケースまで、APIコントラクト全体を検証。
