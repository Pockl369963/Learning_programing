## 1. ユースケースと状態遷移 (Behavior & State)
- **主要なフロー（正常系）:**
  1. ユーザーが新規登録またはログインを行う。
  2. メイン画面から「対オセロAI」または「対ランダム」を選択し、対戦を開始する。
  3. ゲーム開始時、APIを通じて「コイントス」を行い、ランダムでユーザーの先手（黒）/後手（白）を決定する。画面上ではオセロの駒を投げる演出を表示する。
  4. ユーザーが盤面に石を置く（クライアントからAPIへ手番を送信）。
  5. サーバー側で相手（AIまたはランダム）の手を計算し、新しい盤面を返す。
  6. 決着がついた時点で、システムが自動的に勝敗（Win/Loss/Draw）を判定し、対戦履歴をデータベースに保存する。
  7. ユーザーはマイページ等で自分の「直近の対戦履歴（最大10戦）」と勝敗を確認できる。履歴データはAPIによる非同期取得（Fetch API）とし、ゲーム終了直後などでもページ遷移なしにシームレスに更新可能とする。
- **状態遷移:**
  - `UserStatus`: `未ログイン` -> `ログイン済`
  - `GameStatus`: `対戦中 (playing)` -> `決着 (finished)`
- **異常系・エッジケース:**
  - 未ログインユーザーが対戦画面や履歴画面にアクセスした場合は、ログイン画面へリダイレクトする。
  - **途中離脱対応 (放棄判定の拡張):** ブラウザのタブを閉じるなどしてゲームを途中離脱し、新たに次のゲームを開始した際、以前のゲームは強制的に「Loss（敗北）」として扱う。これに加え、以下のルールで放棄（abandoned）とみなす。
    - **「投了（サレンダー）」ボタン**: ユーザーが自ら「このゲームは諦める」ボタンを押した時に放棄とみなし、即座に敗北とする。
    - **タイムアウト（時間切れ）**: 「最後の操作から1時間経過したら、自動的に放棄されたとみなして敗北にする」という定期処理（バッチ処理）をサーバー側で回す。
  - **排他制御（ダブルクリック対策）:** リクエスト時のユーザーの手番（黒or白）と、DBの `current_turn` が一致しない場合（例：通信遅延中に何度もクリックした等）は、楽観的ロックの仕組みとして `409 Conflict` (または `400 Bad Request`) エラーとして弾き、盤面を更新しない。
  - ルール上置けない場所に石を置こうとした場合（不正なAPIリクエスト）は `400 Bad Request` とエラーメッセージを返す。
  - AIモデルの推論に失敗（タイムアウトやエラー）した場合は、エラーをログに記録し、ユーザーにはサーバーエラーとして再試行を促す。

## 3. インターフェース設計 (Contracts)

### APIスキーマの定義 (REST API形式)
対戦中の盤面更新やゲーム初期化は、画面遷移を伴わずスムーズにプレイできるよう非同期通信（Fetch API）で行う。

#### 盤面 (Board) のデータ型定義
フロントエンドとバックエンド間でやり取りする盤面データ `board` （`initial_board` や `next_board`）は、常に **8x8の二次元配列（数値型）** とし、以下の厳密な値を持つものとします。
- `0`: 空のマス
- `1`: 黒（先手）の石
- `-1`: 白（後手）の石

#### `POST /api/game/start/`
ゲーム開始時にコイントス（ランダム）を行い、先手・後手を決定するリクエスト。

- **Request (JSON):**
  ```json
  {
    "opponent_type": "ai"  // "ai" または "random"
  }
  ```

- **Response (JSON):**
  ```json
  {
    "status": "success",
    "session_id": "uuid-string", // 以降のAPIリクエストで利用するゲーム固有のID
    "user_color": 1, // 1: 黒(先手), -1: 白(後手) コイントスの結果
    "opponent_color": -1,
    "initial_board": [[0, 0, ...], ...] // 初期状態。ユーザーが後手(白)の場合、AIの初手が既に打たれた盤面を返すことも可能
  }
  ```

#### `POST /api/game/move/`
プレイヤーが石を置いた時のリクエスト。この1回の通信で、プレイヤーの手の処理と、相手（AI/ランダム）の打ち手を両方処理して結果を返す。双方が置けない「パス（Pass）」の場合もこのエンドポイントを利用する。

- **Request (JSON):**
  ```json
  {
    "session_id": "uuid-string", // /api/game/start/ で発行されたID
    "player_move": {"row": 3, "col": 2}, // ユーザーが石を置く座標。パスの場合は null
    "is_pass": false // プレイヤーがパスする場合は true
  }
  ```

- **Response (JSON):**
  ```json
  {
    "status": "success",
    "next_board": [[0, 0, ...], ...], // プレイヤーと相手の手が両方反映された後の盤面
    "opponent_move": {"row": 2, "col": 2}, // 相手がどこに打ったか（UIのハイライト用）。相手もパスの場合は null
    "opponent_passed": false, // 相手がパスした場合は true
    "is_finished": false, // ゲームが終了したか（盤面が埋まる、または双方がパスした場合 true）
    "result": null, // 終了時のみ "win", "loss", "draw" のいずれかが入る
    "score": null // 終了時のみ {"user": 34, "opponent": 30} のような石の数が入る
  }
  ```

#### `POST /api/game/surrender/`
プレイヤーが「投了（サレンダー）」ボタンを押した際のリクエスト。進行中のゲームを放棄（Loss）扱いとする。

- **Request (JSON):**
  ```json
  {
    "session_id": "uuid-string"
  }
  ```

- **Response (JSON):**
  ```json
  {
    "status": "success",
    "result": "loss"
  }
  ```

#### `GET /api/user/history/`
マイページ等で表示するための、直近の対戦履歴（最大10件）を取得する非同期API。

- **Response (JSON):**
  ```json
  {
    "status": "success",
    "history": [
      {
        "id": 10,
        "opponent_type": "ai",
        "result": "win",
        "played_at": "2023-10-01T12:34:56Z"
      }
      // ... 最大10件
    ]
  }
  ```

#### 共通エラーレスポンス (異常系API)
排他制御エラー（`409 Conflict`）や、不正な手（`400 Bad Request`）などの異常系APIレスポンスのスキーマ。

- **Response (JSON):**
  ```json
  {
    "status": "error",
    "code": "invalid_move", // エラー識別子 (例: "invalid_move", "conflict_turn", "not_authenticated")
    "message": "置けない場所です。" // フロントエンドでユーザーに直接表示可能なメッセージ
  }
  ```

## 4. 非機能要件・運用要件 (NFRs & Observability)
- **パフォーマンス:** AI推論による待ち時間を最小限にするため、レスポンスタイムは通常1秒以内を目指す。
- **セキュリティ:** 
  - **CSRFトークンの受け渡し契約:** 共通レイアウト (`base.html`) の `<head>` 内に `<meta name="csrf-token" content="{{ csrf_token }}">` を埋め込む。JS側（Fetch API）ではこのメタタグからトークンを読み取り、全てのリクエストのヘッダーに `X-CSRFToken: <token>` として付与するルールを徹底する。
  - APIエンドポイントはログインユーザーのみアクセス可能とする（`@login_required`）。
- **オブザーバビリティ (監視・ログ):**
  - アプリケーションエラーや推論失敗などの例外発生時は、Djangoのログ機能によりスタックトレースを出力し、原因特定可能な状態にしておく。
