## 1. プロダクト概要とビジネス目的 (Why)
- **目的:** ユーザーがアカウントを作成して継続的に対戦を楽しめるオセロWebアプリケーションを構築する。
- **主要アクター:** 一般のプレイヤー（ユーザー）
- **スコープ外 (What not to build):** 
  - ランキング機能や他のプレイヤーとのオンライン対戦（スコープを対AI・対ランダムに絞る）
  - 複雑なユーザープロフィール機能、SNS連携
  - パスワードリセット等の高度な認証管理（今回は基本的なユーザーID/パスワードでの登録・ログインのみ）

## 2. 技術スタックの選定
- **バックエンド:** Python (>=3.12), Django (==5.2)
- **データベース:** SQLite (要件指定)
- **AI / 機械学習:** PyTorch (==2.9.1), NumPy (==2.3.5)
- **パッケージマネージャー:** uv
- **静的解析 / フォーマッタ:** Ruff
- **型チェック:** mypy (Django用に django-stubs も併用)
- **セキュリティ監査:** Bandit, pip-audit
- **バックエンドテスト:** pytest, pytest-django
- **フロントエンド:** Django Templates, TailwindCSS (CDNではなくVite等のビルドツールを導入しプロダクト用に最適化), Vanilla JS (オセロ盤面描画・API通信)
- **フロントエンドテスト (E2E):** Playwright (実際の描画やコイントスのアニメーション等のUX、およびAPI通信を包括して担保する)

## 2.5 盤面状態の管理アーキテクチャ (Stateful / サーバー側検証)
オセロの盤面状態（進行中のゲーム）および手の合法性検証は、**サーバー側で管理（Statefulアーキテクチャ）**する方針で確定とします。

- **概要:** ゲーム開始時 (`/api/game/start/`) にDB等（今回は `GameSession` モデル）に「進行中の対戦状態」レコードを作成します。毎ターンのAPI通信時には「どのゲームか（session_id）」と「どこに打ったか（player_move）」だけをクライアントから送信し、サーバー側で盤面状態を保持・更新して手の合法性を検証します。
- **採用理由:**
  1. クライアントからの不正な盤面送信や他人の手を送るなどのチートを完全に防ぐことができるため。
  2. ゲーム開始時にステータスを`playing`で保存するため、次回アクセス時などに「終わっていないゲーム」があれば強制的にLoss（離脱扱い）にするという「途中離脱検知」の要件を厳密に満たすことができるため。

## 4. アーキテクチャ・レイヤー分割の合意
- **Presentation Layer (View/Template):** ユーザーインターフェース。Djangoの **CBV（Class-Based Views: クラスベースビュー）** を全面的に採用し、再利用性と可読性を高める。画面の初期描画はDjango Templateで行い、対戦中の盤面の更新はJavaScriptからREST APIを叩いて非同期に画面を書き換えるSPAライクな挙動とする。
- **Business Logic Layer (Service):** AIモデルへの入力・推論・出力の変換、ランダム手の生成、勝敗判定、対戦履歴の保存制御。
- **Data Access Layer (Model):** SQLiteとのデータのやり取り。Django ORMを利用する。
