# タスク

# 技術スタック
- Python >= 3.12
- Django == 5.2
- pytest, pytest-django, pytest-mock

# 実装対象の機能要件 
以下の3つのビジネスロジックに対するテスト（正常系・異常系・エッジケース）を作成してください。

1. **ゲーム開始とコイントス (`start_game` 関数等)**
   - ユーザーと対戦相手（ai または random）を引数に取る。
   - ランダムに先手（黒: 1）か後手（白: -1）を決定し、`GameSession` レコードを `status='playing'` で作成する。
   - ※テスト時は `random.choice` 等をモック化し、先手・後手の両方のケースを検証すること。

2. **途中離脱の検知と敗北記録 (`handle_abandoned_game` 関数等)**
   - 対象ユーザーの `status='playing'` となっている `GameSession` を `abandoned` に更新する。
   - 上記の更新が行われた場合、連動して `MatchHistory` に `result='loss'` として履歴が自動的に記録されること。

3. **最新10件の履歴保持トランザクション (`save_match_history` 関数等)**
   - 対戦履歴を保存する際、対象ユーザーの履歴が11件以上になった場合、最も古いレコードが物理削除され、常に最新10件のみが保持されること。
   - この処理は `transaction.atomic` によってアトミックに行われることを前提とする（テスト内でレコード作成後に総数が10件を超えないことをアサーションする）。
   - ちょうど10件の場合、11件になる場合などの境界値テスト（エッジケース）を含めること。

# テスト実装における制約
1. `@pytest.mark.django_db` デコレータを使用し、DBにアクセスするテストとして記述すること。
2. AI推論モジュール（`model/agent.py` や `model/othello_env.py` を呼び出す部分）がもし初期化ロジックに含まれる場合は、`mocker.patch` を使用して完全にモック化すること。AIの実際の計算はテストで実行させない。
3. `django.contrib.auth.models.User` を使用して、テスト用のダミーユーザーをセットアップするフィクスチャ (`pytest.fixture`) を用意すること。
4. アサーションは明確に行い、Arrange-Act-Assert (AAA) パターンで可読性高く記述すること。

それでは、`tests/test_services.py` のコードを出力してください。
